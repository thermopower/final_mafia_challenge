# 제품 요구사항 문서 (Product Requirements Document)

## 프로젝트 개요

### 프로젝트 명
대학교 사내 데이터 시각화 대시보드 (University Data Visualization Dashboard)

### 프로젝트 버전
v1.0.0

### 작성일
2025년 11월 1일

### 프로젝트 배경
대학교 내부에서 사용하는 실적, 논문 게재 수, 학생 수, 예산 등의 데이터는 현재 이카운트(Ecount) 시스템을 통해 관리되고 있습니다. 그러나 기본 제공되는 그래프의 시인성이 낮고 활용도가 떨어져 데이터 기반 의사결정에 어려움을 겪고 있습니다. 본 프로젝트는 이러한 문제를 해결하기 위해 Ecount에서 추출한 데이터(엑셀)를 기반으로, 사용자가 각 항목의 성과와 추이를 한눈에 파악할 수 있는 맞춤형 그래프 및 차트를 제공하는 웹 기반 대시보드를 구축하는 것을 목표로 합니다.

### 프로젝트 목표
1. **데이터 시각화 향상**: 복잡한 데이터를 직관적인 차트와 그래프로 변환하여 시인성 개선
2. **의사결정 지원**: 실시간으로 업데이트되는 데이터 기반의 인사이트 제공
3. **효율성 증대**: 엑셀 파일 업로드만으로 자동화된 데이터 처리 및 시각화
4. **접근성 보장**: 내부 직원들이 웹 브라우저를 통해 언제 어디서나 접근 가능
5. **확장성 확보**: 향후 추가적인 데이터 소스 및 시각화 기능 확장 가능한 아키텍처

### 비즈니스 가치
- **시간 절약**: 수동 보고서 작성 시간을 80% 이상 감소
- **정확성 향상**: 데이터 입력 오류 최소화 및 자동 검증
- **인사이트 발견**: 데이터 트렌드 및 패턴 조기 발견
- **자원 최적화**: 데이터 기반 예산 배분 및 자원 관리

## Stakeholders (이해관계자)

### 1. Primary Stakeholders (주요 이해관계자)

#### 1.1 대학교 경영진
- **역할**: 최종 의사결정자, 전략적 데이터 소비자
- **관심사항**:
  - 대학 전체의 성과 지표 (KPI) 한눈에 파악
  - 학과별, 단과대학별 비교 분석
  - 예산 집행 현황 및 효율성
  - 연구 성과 및 논문 게재 추이
- **성공 기준**: 의사결정에 필요한 정보를 5분 이내에 파악 가능

#### 1.2 행정 담당자 (관리자)
- **역할**: 시스템 관리자, 데이터 업로드 담당자
- **관심사항**:
  - 엑셀 파일 업로드 용이성
  - 데이터 파싱 오류 발생 시 명확한 피드백
  - 사용자 계정 관리
  - 시스템 안정성 및 신뢰성
- **성공 기준**: 주간 데이터 업데이트 시간 30분 이내

#### 1.3 학과장/단과대학장
- **역할**: 중간 관리자, 부서별 성과 관리자
- **관심사항**:
  - 소속 학과/단과대학의 성과 지표
  - 타 학과/단과대학과의 비교
  - 취업률, 연구비, 논문 게재 수 등 세부 지표
- **성공 기준**: 월간 보고서 작성 시간 50% 단축

#### 1.4 연구 지원팀
- **역할**: 연구 과제 관리, 예산 집행 모니터링
- **관심사항**:
  - 연구 과제별 예산 집행 현황
  - 과제별 논문 산출 연계 정보
  - 기술이전 수입 추이
- **성공 기준**: 연구 과제 모니터링 효율성 70% 향상

### 2. Secondary Stakeholders (2차 이해관계자)

#### 2.1 IT 부서
- **역할**: 시스템 유지보수, 기술 지원
- **관심사항**:
  - 시스템 안정성 및 성능
  - 보안 및 접근 제어
  - 백업 및 복구 절차
- **성공 기준**: 시스템 가동률 99.5% 이상

#### 2.2 감사팀
- **역할**: 데이터 정확성 검증, 규정 준수 확인
- **관심사항**:
  - 데이터 추적 가능성 (audit trail)
  - 데이터 무결성
  - 접근 로그 관리
- **성공 기준**: 모든 데이터 변경 이력 100% 추적 가능

#### 2.3 개발팀 (외주 개발사)
- **역할**: 시스템 개발 및 구축
- **관심사항**:
  - 명확한 요구사항 정의
  - 기술 스택 선정의 합리성
  - 유지보수 용이성
- **성공 기준**: 기한 내 품질 기준을 만족하는 시스템 납품

## 포함 페이지 (Page Structure)

### 1. 인증 페이지

#### 1.1 로그인 페이지 (Login Page)
- **경로**: `/login`
- **접근 권한**: 비로그인 사용자
- **주요 기능**:
  - 이메일/비밀번호 기반 로그인
  - 로그인 실패 시 오류 메시지 표시
  - "비밀번호 찾기" 링크 (선택 사항)
  - 자동 로그인 유지 옵션 (Remember Me)
- **UI 요소**:
  - 이메일 입력 필드
  - 비밀번호 입력 필드
  - 로그인 버튼
  - 에러 메시지 영역

### 2. 메인 대시보드 페이지

#### 2.1 통합 대시보드 (Main Dashboard)
- **경로**: `/dashboard` 또는 `/`
- **접근 권한**: 로그인된 모든 사용자
- **주요 기능**:
  - 핵심 성과 지표 (KPI) 카드 표시
  - 연도별 성과 추이 차트
  - 학과별/단과대학별 비교 차트
  - 필터링 및 기간 선택 기능
- **표시 데이터**:
  - 전임교원 수 총계 및 추이
  - 초빙교원 수 총계 및 추이
  - 졸업생 취업률 평균 및 추이
  - 기술이전 수입액 총계 및 추이
  - 국제학술대회 개최 횟수
  - 논문 게재 수 (SCIE/KCI 분류)
  - 학생 수 (학사/석사/박사 분류)
  - 연구 과제 예산 집행률
- **차트 유형**:
  - KPI 메트릭 카드 (Metric Cards)
  - 막대 그래프 (Bar Chart) - 학과별 비교
  - 라인 차트 (Line Chart) - 연도별 추이
  - 파이 차트 (Pie Chart) - 비율 표시
  - 테이블 (Table) - 상세 데이터
- **필터링 옵션**:
  - 연도 선택 (2023, 2024, 2025)
  - 단과대학 선택 (전체, 공과대학, 인문대학, 사범대학 등)
  - 학과 선택

#### 2.2 연구 성과 대시보드 (Research Performance Dashboard)
- **경로**: `/dashboard/research`
- **접근 권한**: 로그인된 모든 사용자
- **주요 기능**:
  - 논문 게재 현황 시각화
  - 연구 과제 진행 현황
  - 기술이전 수입 추이
- **표시 데이터**:
  - 학술지 등급별 논문 수 (SCIE/KCI)
  - Impact Factor 평균
  - 과제 연계 논문 비율
  - 연구비 집행 현황
  - 학과별 연구 성과 비교
- **차트 유형**:
  - 스택 바 차트 (Stacked Bar Chart) - 등급별 논문 수
  - 라인 차트 - 연도별 논문 수 추이
  - 테이블 - 논문 상세 목록

#### 2.3 학생 현황 대시보드 (Student Dashboard)
- **경로**: `/dashboard/students`
- **접근 권한**: 로그인된 모든 사용자
- **주요 기능**:
  - 학생 수 통계
  - 학적 상태 분포
  - 학과별 학생 분포
- **표시 데이터**:
  - 총 학생 수
  - 과정별 학생 수 (학사/석사/박사)
  - 학적 상태별 분포 (재학/휴학/졸업)
  - 성별 분포
  - 학과별 학생 수
- **차트 유형**:
  - 파이 차트 - 과정별/학적상태별 분포
  - 막대 그래프 - 학과별 학생 수
  - 테이블 - 학생 목록

#### 2.4 예산 집행 대시보드 (Budget Execution Dashboard)
- **경로**: `/dashboard/budget`
- **접근 권한**: 관리자, 연구 지원팀
- **주요 기능**:
  - 연구 과제별 예산 집행 현황
  - 집행 항목별 분석
  - 지원 기관별 통계
- **표시 데이터**:
  - 총 연구비 대비 집행액
  - 집행 완료/처리 중 비율
  - 집행 항목별 금액 (인건비, 장비비, 재료비 등)
  - 지원 기관별 연구비 현황
- **차트 유형**:
  - 프로그레스 바 (Progress Bar) - 집행률
  - 파이 차트 - 집행 항목별 비율
  - 라인 차트 - 월별 집행 추이
  - 테이블 - 집행 상세 내역

## 사용자 여정 (User Journey)

### Journey 1: 관리자 - 데이터 업로드 및 대시보드 확인

#### 타겟 유저 Segment
- **페르소나**: 김민정 (행정팀 대리, 30세)
- **역할**: 데이터 관리 담당자
- **목표**: 주간 데이터 업데이트를 빠르고 정확하게 완료
- **Pain Point**: 기존 시스템은 업로드 오류가 잦고 오류 원인을 찾기 어려움

#### 단계별 여정

**1단계: 로그인**
- **페이지**: `/login`
- **행동**:
  - 브라우저에서 대시보드 URL 접속
  - 이메일과 비밀번호 입력
  - "로그인" 버튼 클릭
- **기대 결과**: 로그인 성공 후 메인 대시보드로 자동 이동
- **감정**: 안정적 (로그인이 빠르고 간단함)

**2단계: 데이터 업로드 페이지 이동**
- **페이지**: `/dashboard` → `/upload`
- **행동**:
  - 상단 네비게이션 바에서 "데이터 업로드" 메뉴 클릭
- **기대 결과**: 데이터 업로드 페이지 로드
- **감정**: 확신 (명확한 메뉴 구조)

**3단계: 엑셀 파일 업로드**
- **페이지**: `/upload`
- **행동**:
  - 데이터 타입 선택 (예: "학과 KPI 데이터")
  - Ecount에서 다운로드한 엑셀 파일을 드래그 앤 드롭
  - "업로드" 버튼 클릭
- **기대 결과**:
  - 업로드 진행률 표시
  - 파싱 완료 후 데이터 미리보기 표시
  - 성공 메시지: "12개 행이 성공적으로 업로드되었습니다."
- **감정**: 만족 (진행 상황이 명확하게 보임)

**3-1단계: 오류 발생 시**
- **페이지**: `/upload`
- **행동**:
  - 오류 메시지 확인
  - 예: "3행: '졸업생 취업률' 컬럼이 숫자가 아닙니다. 값: '85.5%'"
  - 엑셀 파일 수정 후 재업로드
- **기대 결과**: 명확한 오류 위치와 원인 안내
- **감정**: 안도 (문제 해결 방법이 명확함)

**4단계: 대시보드 확인**
- **페이지**: `/upload` → `/dashboard`
- **행동**:
  - 상단 네비게이션 바에서 "대시보드" 메뉴 클릭
  - 업로드한 데이터가 차트에 반영되었는지 확인
  - 필터를 사용해 특정 학과 데이터 확인
- **기대 결과**:
  - 최신 데이터가 실시간으로 차트에 반영됨
  - 연도별 추이가 명확하게 보임
- **감정**: 만족 (데이터가 즉시 반영되고 시각화가 직관적)

**5단계: 로그아웃**
- **페이지**: `/dashboard`
- **행동**:
  - 우측 상단 프로필 아이콘 클릭
  - "로그아웃" 버튼 클릭
- **기대 결과**: 로그인 페이지로 이동
- **감정**: 완료감 (업무가 빠르게 마무리됨)

#### 예상 소요 시간
- 전체 여정: 약 5-10분
- 핵심 KPI: 데이터 업로드 성공률 95% 이상

### Journey 2: 경영진 - 월간 성과 보고서 작성을 위한 데이터 조회

#### 타겟 유저 Segment
- **페르소나**: 박준호 (기획처장, 52세)
- **역할**: 대학 경영 의사결정자
- **목표**: 월간 경영 회의 자료 준비를 위한 핵심 지표 파악
- **Pain Point**: 기존에는 여러 엑셀 파일을 열어 수동으로 비교해야 했음

#### 단계별 여정

**1단계: 로그인**
- **페이지**: `/login`
- **행동**: 이메일과 비밀번호로 로그인
- **기대 결과**: 메인 대시보드로 이동
- **감정**: 익숙함 (주 1-2회 접속)

**2단계: 통합 대시보드 조회**
- **페이지**: `/dashboard`
- **행동**:
  - KPI 카드에서 주요 지표 확인:
    - 전체 논문 게재 수 증감률
    - 평균 취업률
    - 총 연구비 집행률
  - 연도 필터를 "2024" → "2025"로 변경하여 증감 추이 확인
- **기대 결과**:
  - 한눈에 핵심 지표 파악
  - 전년 대비 증감률 표시 (예: +15% ↑)
- **감정**: 효율적 (정보 파악이 빠름)

**3단계: 연구 성과 상세 분석**
- **페이지**: `/dashboard` → `/dashboard/research`
- **행동**:
  - 사이드바에서 "연구 성과" 클릭
  - SCIE/KCI 논문 수 비교 차트 확인
  - 학과별 논문 수 막대 그래프 확인
  - 상위 3개 학과 식별 (컴퓨터공학과, 전자공학과 등)
- **기대 결과**:
  - 학과별 성과 차이가 명확하게 보임
  - 우수 학과와 개선 필요 학과 식별
- **감정**: 인사이트 획득 (데이터 기반 판단 가능)

**4단계: 예산 집행 현황 점검**
- **페이지**: `/dashboard/research` → `/dashboard/budget`
- **행동**:
  - 사이드바에서 "예산 집행" 클릭
  - 총 연구비 대비 집행률 확인 (예: 67%)
  - 지연되고 있는 과제 식별 (처리 중 상태)
  - 집행 항목별 비율 파이 차트 확인
- **기대 결과**:
  - 예산 집행 지연 과제 파악
  - 집행 항목별 비율 확인
- **감정**: 경각심 (일부 과제 집행 지연 발견)

**5단계: 데이터 엑셀 내보내기 (선택 사항)**
- **페이지**: `/data/view`
- **행동**:
  - "데이터 조회" 메뉴 클릭
  - "논문 목록" 탭 선택
  - 필터 적용 (2025년, SCIE)
  - "엑셀 다운로드" 버튼 클릭
- **기대 결과**: 필터링된 데이터가 엑셀 파일로 다운로드됨
- **감정**: 편리함 (보고서 작성에 바로 활용 가능)

**6단계: 로그아웃**
- **페이지**: 프로필 메뉴에서 로그아웃
- **기대 결과**: 로그인 페이지로 이동
- **감정**: 만족 (필요한 정보를 빠르게 얻음)

#### 예상 소요 시간
- 전체 여정: 약 10-15분
- 핵심 KPI: 보고서 작성 시간 50% 단축

### Journey 3: 학과장 - 학과 성과 모니터링

#### 타겟 유저 Segment
- **페르소나**: 이서연 교수 (컴퓨터공학과 학과장, 45세)
- **역할**: 학과 운영 책임자
- **목표**: 학과 성과 추이 파악 및 타 학과와의 비교
- **Pain Point**: 학과 성과를 정기적으로 모니터링할 수 있는 도구 부재

#### 단계별 여정

**1단계: 로그인**
- **페이지**: `/login`
- **행동**: 로그인
- **기대 결과**: 대시보드로 이동
- **감정**: 기대 (학과 성과 확인)

**2단계: 학과 필터 적용**
- **페이지**: `/dashboard`
- **행동**:
  - 필터 옵션에서 "학과" 선택
  - "컴퓨터공학과" 선택
  - "적용" 버튼 클릭
- **기대 결과**:
  - 모든 차트가 컴퓨터공학과 데이터로 필터링됨
  - KPI 카드에 학과 전용 지표 표시
- **감정**: 집중 (자신의 학과 데이터에 집중)

**3단계: 연구 성과 확인**
- **페이지**: `/dashboard/research`
- **행동**:
  - 논문 게재 수 추이 라인 차트 확인
  - 2023년: 2편 → 2024년: 4편 → 2025년: 6편 (추정)
  - SCIE 논문 비율 확인
- **기대 결과**:
  - 연구 성과가 지속적으로 증가하고 있음을 확인
  - 목표 대비 진행 상황 파악
- **감정**: 자부심 (성과 향상 확인)

**4단계: 타 학과와 비교**
- **페이지**: `/dashboard/research`
- **행동**:
  - 필터를 "전체 학과"로 변경
  - 학과별 논문 수 막대 그래프 확인
  - 컴퓨터공학과가 3위인 것 확인
- **기대 결과**:
  - 상위 학과와의 격차 파악
  - 개선 필요 영역 식별
- **감정**: 동기 부여 (목표 설정)

**5단계: 학생 현황 확인**
- **페이지**: `/dashboard/students`
- **행동**:
  - 학과 필터를 다시 "컴퓨터공학과"로 변경
  - 학사/석사 학생 수 확인
  - 휴학 학생 비율 확인
- **기대 결과**:
  - 학생 구성 파악
  - 휴학률이 높으면 원인 분석 필요성 인지
- **감정**: 관심 (학생 복지 개선 필요성 인지)

**6단계: 로그아웃**
- **페이지**: 로그아웃
- **기대 결과**: 로그인 페이지로 이동
- **감정**: 만족 (정기 모니터링 완료)

#### 예상 소요 시간
- 전체 여정: 약 8-12분
- 핵심 KPI: 주간 모니터링 소요 시간 70% 단축

│       └── 검색 및 필터
│           ├── 이름/이메일 검색

│
└── 공통 컴포넌트 (Global Components)
    │
    ├── 네비게이션 바 (모든 페이지)
    │   ├── 로고/제목

        └── 지원 연락처

### IA 설계 원칙

1. **명확한 계층 구조**: 3단계 이내의 깊이로 제한하여 사용자가 원하는 정보에 빠르게 도달
2. **일관성**: 모든 페이지에서 동일한 네비게이션 패턴 유지
3. **권한 기반 접근**: 관리자 전용 기능은 명확히 구분하여 표시
4. **유연한 필터링**: 모든 대시보드에서 일관된 필터 인터페이스 제공
5. **검색 가능성**: 데이터 조회 페이지에서 전체 텍스트 검색 지원

## 핵심 기능 요구사항 (Core Functional Requirements)

### 1. 인증 및 권한 관리 (Authentication & Authorization)

#### 1.1 사용자 인증
- **FR-AUTH-001**: 시스템은 이메일과 비밀번호 기반 로그인을 지원해야 함
- **FR-AUTH-002**: 로그인 실패 시 명확한 오류 메시지를 표시해야 함 (예: "이메일 또는 비밀번호가 올바르지 않습니다")
- **FR-AUTH-003**: JWT 토큰 기반 인증을 사용하여 API 요청을 보호해야 함
- **FR-AUTH-004**: 토큰 만료 시 자동으로 재인증 또는 로그인 페이지로 리디렉션해야 함
- **FR-AUTH-005**: "자동 로그인 유지" 옵션을 제공하여 30일간 세션 유지 가능

#### 1.2 권한 관리
- **FR-AUTH-006**: 시스템은 최소 2가지 권한 레벨을 지원해야 함:
  - **관리자 (Admin)**: 모든 기능 접근 가능 (데이터 업로드, 사용자 관리, 예산 대시보드 등)
  - **일반 사용자 (User)**: 대시보드 조회 및 데이터 조회만 가능
- **FR-AUTH-007**: 권한이 없는 페이지 접근 시 403 에러 페이지 또는 알림을 표시해야 함
- **FR-AUTH-008**: 관리자는 사용자 관리 페이지에서 다른 사용자의 권한을 변경할 수 있어야 함

#### 1.3 사용자 관리 (관리자 전용)
- **FR-AUTH-009**: 관리자는 새 사용자를 추가할 수 있어야 함 (이메일, 이름, 소속 부서, 권한)
- **FR-AUTH-010**: 관리자는 임시 비밀번호를 생성하여 사용자에게 전달할 수 있어야 함
- **FR-AUTH-011**: 관리자는 사용자 계정을 활성화/비활성화할 수 있어야 함
- **FR-AUTH-012**: 관리자는 사용자 목록을 조회하고 검색할 수 있어야 함

### 2. 데이터 업로드 및 파싱 (Data Upload & Parsing)

#### 2.1 파일 업로드
- **FR-UPLOAD-001**: 시스템은 .xlsx 및 .csv 형식의 엑셀 파일 업로드를 지원해야 함
- **FR-UPLOAD-002**: 파일 크기 제한은 최대 10MB로 설정해야 함
- **FR-UPLOAD-003**: 드래그 앤 드롭 및 파일 선택 버튼 두 가지 업로드 방식을 지원해야 함
- **FR-UPLOAD-004**: 업로드 전에 데이터 타입을 선택해야 함 (학과 KPI, 논문, 연구 과제, 학생)
- **FR-UPLOAD-005**: 업로드 진행 상태를 프로그레스 바로 표시해야 함

#### 2.2 데이터 파싱 및 검증
- **FR-UPLOAD-006**: 시스템은 업로드된 엑셀 파일의 컬럼 헤더를 검증해야 함
  - 예: 학과 KPI 데이터 → "평가년도, 단과대학, 학과, 졸업생 취업률 (%)" 등 필수 컬럼 존재 확인
- **FR-UPLOAD-007**: 각 컬럼의 데이터 타입을 검증해야 함
  - 예: "졸업생 취업률 (%)"는 숫자여야 함, "%" 기호가 포함되면 제거 후 숫자로 변환
- **FR-UPLOAD-008**: 중복 데이터 검사를 수행해야 함
  - 예: 동일한 (연도, 학과) 조합이 이미 존재하면 업데이트 또는 오류 표시
- **FR-UPLOAD-009**: 파싱 오류 발생 시 다음 정보를 포함한 오류 메시지를 표시해야 함:
  - 행 번호
  - 컬럼 이름
  - 오류 내용
  - 예: "3행: '졸업생 취업률' 컬럼이 숫자가 아닙니다. 값: '85.5%'"
- **FR-UPLOAD-010**: 파싱 성공 시 데이터 미리보기 테이블을 표시해야 함 (최대 100행)

#### 2.3 데이터 저장
- **FR-UPLOAD-011**: 검증된 데이터를 PostgreSQL 데이터베이스에 저장해야 함
- **FR-UPLOAD-012**: 데이터 저장 성공 시 "N개 행이 성공적으로 업로드되었습니다" 메시지를 표시해야 함
- **FR-UPLOAD-013**: 업로드 이력을 기록해야 함 (업로드 날짜, 파일명, 데이터 타입, 행 수, 업로드자)

#### 2.4 지원 데이터 타입

**학과 KPI 데이터 (department_kpi)**
- 필수 컬럼: 평가년도, 단과대학, 학과, 졸업생 취업률 (%), 전임교원 수 (명), 초빙교원 수 (명), 연간 기술이전 수입액 (억원), 국제학술대회 개최 횟수
- 데이터 타입: 평가년도(정수), 단과대학(문자열), 학과(문자열), 나머지(숫자)

**논문 목록 (publication_list)**
- 필수 컬럼: 논문ID, 게재일, 단과대학, 학과, 논문제목, 주저자, 참여저자, 학술지명, 저널등급, Impact Factor, 과제연계여부
- 데이터 타입: 게재일(날짜), Impact Factor(숫자), 과제연계여부(Y/N), 나머지(문자열)

**연구 과제 데이터 (research_project_data)**

- 데이터 타입: 집행일자(날짜), 총연구비(정수), 집행금액(정수), 나머지(문자열)

**학생 명단 (student_roster)**
- 필수 컬럼: 학번, 이름, 단과대학, 학과, 학년, 과정구분, 학적상태, 성별, 입학년도, 지도교수, 이메일
- 데이터 타입: 학번(문자열), 학년(정수), 입학년도(정수), 나머지(문자열)

## 비기능 요구사항 (Non-Functional Requirements)

### 1. 성능 (Performance)

#### 1.1 응답 시간
- **NFR-PERF-001**: 대시보드 페이지 로드 시간은 3초 이내여야 함
- **NFR-PERF-002**: API 응답 시간은 평균 500ms 이내여야 함
- **NFR-PERF-003**: 차트 렌더링 시간은 1초 이내여야 함
- **NFR-PERF-004**: 엑셀 파일 업로드 및 파싱 시간은 10MB 파일 기준 30초 이내여야 함

#### 1.2 처리량
- **NFR-PERF-005**: 시스템은 최소 50명의 동시 사용자를 지원해야 함
- **NFR-PERF-006**: 데이터베이스는 최소 100만 행의 데이터를 처리할 수 있어야 함

#### 1.3 확장성
- **NFR-PERF-007**: 시스템은 수평 확장(horizontal scaling)이 가능해야 함
- **NFR-PERF-008**: 데이터베이스는 인덱싱을 통해 쿼리 성능을 최적화해야 함

### 2. 보안 (Security)

#### 2.1 인증 및 권한
- **NFR-SEC-001**: 모든 API 엔드포인트는 JWT 토큰 기반 인증을 요구해야 함
- **NFR-SEC-002**: 비밀번호는 bcrypt 또는 Argon2 해싱 알고리즘으로 저장해야 함
- **NFR-SEC-003**: JWT 토큰 만료 시간은 24시간으로 설정해야 함
- **NFR-SEC-004**: Refresh Token을 사용하여 세션 연장을 지원해야 함

#### 2.2 데이터 보호
- **NFR-SEC-005**: HTTPS를 통한 모든 통신을 암호화해야 함
- **NFR-SEC-006**: CORS 정책을 설정하여 허가된 도메인만 API 접근을 허용해야 함
- **NFR-SEC-007**: SQL Injection, XSS 등 일반적인 웹 공격을 방어해야 함

- **알림 기능**: 중요 지표 달성/미달 시 이메일 또는 슬랙 알림
- **고급 필터링**: 다중 조건 필터, 저장된 필터 프리셋
- **데이터 비교**: 연도별, 학과별 성과 비교 분석 페이지
- **예측 분석**: 머신러닝 기반 트렌드 예측 (선택 사항)

### Phase 3 (선택 사항)
- **모바일 앱**: React Native 기반 모바일 앱 개발
- **실시간 협업**: 댓글, 공유, 북마크 기능
- **보고서 자동 생성**: PDF 형식의 월간/분기 보고서 자동 생성
- **다국어 지원**: 영어, 중국어 등 다국어 인터페이스

## 용어 정리 (Glossary)

- **Ecount**: 대학교에서 사용 중인 기존 데이터 관리 시스템
- **KPI (Key Performance Indicator)**: 핵심 성과 지표
- **SCIE**: Science Citation Index Expanded (국제 학술지 등급)
- **KCI**: Korea Citation Index (한국 학술지 등급)
- **Impact Factor**: 학술지의 영향력 지수
- **Supabase**: PostgreSQL 기반의 오픈소스 BaaS (Backend as a Service)
- **Railway**: 클라우드 배포 플랫폼
- **Vercel**: 프론트엔드 배포 플랫폼
- **JWT (JSON Web Token)**: 토큰 기반 인증 방식
- **CORS (Cross-Origin Resource Sharing)**: 교차 출처 리소스 공유
- **ORM (Object-Relational Mapping)**: 객체-관계 매핑
- **UAT (User Acceptance Test)**: 사용자 인수 테스트
- **RTO (Recovery Time Objective)**: 목표 복구 시간
- **RPO (Recovery Point Objective)**: 목표 복구 시점
